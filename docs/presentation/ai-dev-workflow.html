The Problem

Game engine development is expensive

Complexity

Physics, rendering, animation, IK, serialization — each system has dozens of interdependent edge cases.

Consistency

Without enforced standards, architecture drifts. Naming, patterns, and module contracts diverge over time.

Velocity

Writing tests, boilerplate, and documentation for every feature is slow — often skipped under pressure.

The bottleneck isn't ideas. It's the cost of translating ideas into tested, documented, shippable code.
The Solution

Treat AI as a senior engineer on the team

Spec-driven design

Every feature starts with a structured spec — problem, solution, acceptance criteria, traceability — before a line of code is written.

AI-assisted implementation

Claude reads the spec, the parent contracts, and the codebase — then generates code that respects every binding decision.

Research first

Ambiguous problems get a structured research funnel: explore → ideate → evaluate → choose — producing a defensible decision record.

Test generation

Comprehensive tests (unit, stress, golden, invariant, determinism, integration) generated automatically from the public API.

Architecture

A 4-level spec hierarchy — everything traces up

Every piece of work lives at one of four levels. Each level inherits binding decisions from its parent. Nothing ships without a complete lineage.

Platform — Cluiche (platform-wide constraints, tech decisions)
  └─ Application — Dia Engine (engine-wide contracts)
      └─ System — DiaVisualDebugger, DiaIK2D, DiaAnimation2D …
           └─ Feature — fabrik-solver, bone-lines-drawer, fixed-draw-layer …

Traceability table

Every feature spec has a table linking it back through System → Application → Platform.

Binding decisions

Platform decisions (e.g. "no heap alloc in hot path") are enforced in every child spec's compliance section.

Status gates

Draft → Approved → In Progress → Done. A system is only Done when all child feature specs are Approved.

Spec Anatomy

What a feature spec contains

  • Problem Statement — why this feature exists
  • Solution Overview — numbered algorithm / design
  • Acceptance Criteria — testable pass/fail conditions
  • Binding Decisions Compliance — table showing every Platform/Application constraint is respected
  • AI Review Questions — generated risks, edge cases, integration gaps
  • Traceability — links up to System → App → Platform
  • Status — and link to its plan file

Spec Process

5 mandatory steps before Approval

1
Interview — AI asks structured questions before writing anything. No spec from thin air.
2
Draft — Full spec body: summary, goals, solution overview, tasks, traceability.
3
Binding Decisions — Compliance table against every parent constraint. Never optional.
4
AI Review Questions — Generated risks, gaps, and edge cases — all answered. Never optional.
5
Approval Gate — Explicit confirmation. Only now can implementation start.

Hard rule: if the user says "quickly" or "just get it done" — treat it as a red flag and do all 5 steps anyway.

Implementation

From approved spec to merged code

  • Spec exists + Approved — implementation gate; no approved spec = no code
  • Create a plan — *.plan.md alongside the spec; task table with status, model, notes
  • Read the full chain — AI reads Platform → App → System → Feature before touching code
  • Delegate to subagents — each plan task = one subagent; parallel where independent
  • Commit after each task — atomic commits, plan updated in the same commit
  • Spec status updated — In Progress during work, Done after all tasks pass

Testing

Comprehensive tests generated from the public API

The /gen-tests command reads public API headers, checks coverage registry, and proposes all applicable test types.

Unit

Each public method tested in isolation. Named ComponentName_Method_Scenario_Expected.

Golden Value

Known inputs → known outputs. Catches regressions when algorithms change.

Invariant

Properties that must always hold: root bone pinned, chain length conserved, etc.

Stress / Boundary

Max capacity, zero-length inputs, degenerate geometry — wherever the code might break.

Determinism

Same input → same output on every run. Critical for physics and animation.

Integration

End-to-end: IK solver updates skeleton FK; visual debugger draws correct primitives.

Testing Pattern

Test helpers ship with the library, not the test suite

A key architectural decision: test utilities live in <Module>/Testing/ alongside production code — not buried in GoogleTests.

  • Discoverable — any consumer of the module can use the test builders
  • Reusable — integration tests across modules use the same helpers
  • Maintained together — when the API changes, the test utilities change alongside it
  • No test-only binaries — helpers compile into the library; no runtime cost when not used

  IKSolver.h # production API
  IKSolver.cpp
  Testing/

Debugging

Visual debugging as a first-class system

DiaVisualDebugger provides a layer-managed, priority-sorted debug render pipeline. Toggled via in-game console or editor panel — same DiaAPI command regardless of surface.

  • DebugLayerManager — registry of named draw layers with priority
  • IVisualDebugger — pure interface: reads const sim state, writes primitives
  • DebugColourPalette — canonical RGBA constants with semantic names
  • FixedDrawLayer — register objects once; renderer auto-draws every frame
  • ImGui console — in-game toggle panel + DiaAPI command input + log tail
  • #ifdef DIA_DEBUG — zero overhead in Release builds
// Register a draw layer once

  priority: 10
);

Tooling

Custom slash commands — workflows as first-class tools

Every repeatable workflow is a slash command. Typed in the chat; executes a structured multi-step process with mandatory gates.

  • /spec-feature — interview + 5-step spec process + approval gate
  • /spec-system — system spec with feature inventory
  • /spec-review — populate AI review questions on an existing spec
  • /spec-trace — trace a feature's full lineage to platform
  • /gen-tests <target> — generate all test types from public API
  • /research — structured explore→ideate→evaluate→choose funnel
  • /refactor — audit → plan → execute → review → prove loop
  • /security-review — review pending changes for vulnerabilities

Why commands matter

Commands encode the process, not just the output. You can't skip step 3 of a spec — the command won't let you. The workflow is enforced, not suggested.

New team members run the same commands and get the same quality output, regardless of experience level.

The Takeaway

What makes this a superpower

CLAUDE.md, steering docs, and binding decisions mean the AI doesn't drift. It writes code that matches your patterns — even patterns you set months ago.

Specs prevent rework

Getting Started

How to adopt this in your project

  • Step 1: Write a CLAUDE.md — architecture overview, naming conventions, build commands, code patterns. This is the AI's onboarding doc.
  • Step 2: Create a docs/specs/ hierarchy with a Platform spec and one Application spec. Run /spec-system for your first system.
  • Step 3: Add .claude/steering/tech.md — language, testing standards, git workflow.
  • Step 4: Use /spec-feature for every new feature. No exceptions.
  • Step 5: Use /gen-tests after each implementation. Add it to your definition of Done.
  • Step 6: Use /research whenever a decision isn't clear. Archive the output.

Minimum viable start

You don't need the full system on day one. Start with just:

  • A CLAUDE.md
  • One platform spec
  • Mandatory spec-before-code rule

The rest of the structure emerges as you need it. The key is the spec habit — everything else follows.
