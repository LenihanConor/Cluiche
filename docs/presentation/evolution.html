The Vision

Cluiche — an Irish word for "game"

A production-quality game development platform. Not a prototype, not a tutorial project — a real engine with real architecture, real tests, and real tooling. Built by one developer, with an AI as a senior collaborator on the team.

Platform: Cluiche

9 binding decisions. Every module, every feature traces back to the platform spec. No orphaned code.

Engine: Dia

37+ modules across physics, animation, rigging, geometry, serialization, state machines, debugging, and tooling.

Methodology: Spec-First

Platform → Application → System → Feature. Nothing ships without an approved spec and a traceable lineage.

Phase 1 — Apr 23

The Editor Layer

Day one of tracked development. The goal: a real plugin-based editor with live debug introspection. Built the full scaffolding in one sitting.

  • CluicheEditor — standalone editor application with ImGui docking, panel management, fullscreen mode
  • DiaApplicationEditor plugin — manifest editing UI with lifecycle view, flow view, module inspector
  • Splash screen — native Win32 splash with React overlay, shown during engine startup
  • Protobuf migration — replaced string payloads with google.protobuf.Struct in DiaDebugServer
  • WebSocket lifecycle fix — client-reuse bug tracked and resolved
  • 258 tests — Vitest suites for editor UI, integration tests for plugin pipeline

The pattern established here

Every editor panel is a DiaAPI plugin. The plugin manifest drives the UI — no hardcoded panel layouts. This pattern carries through DiaApplicationEditor, DiaPipelineEditor, and DiaAssetCatalogueEditor.

Claude generated all Vitest test suites from the component interfaces, catching the DockingManager fullscreen bug before it shipped.

Phase 4 — May 1–2

The Simulation Layer

Five systems across state, rigging, IK, animation, and serialization — all built spec-first with full test suites.

DiaStateMachine — 71 tests

Flat FSM, hierarchical state machine (HSM), and pushdown automaton. TransitionRecorder, Clone, HSM 3-level nesting, full tracer modes.

DiaRig2D — 82 tests

2D skeletal rig: ISkeletonLoader, bone metadata, FK, pose blending. 44 unit tests + 38 exhaustive (golden, invariant, stress, boundary, determinism, integration).

DiaIK2D — 33 tests

Post-process IK: two-bone solver, FABRIK (iterative), look-at constraint. No heap allocation during solve. Root bone pinned. Joint limits clamped per iteration.

DiaAnimation2D — 120 tests

Keyframe clip player, pose blend stack, damped spring chains, animation evaluator. 107 initial tests + 13 more after pointer stability bug discovered and fixed.

DiaSerializer Phase 3 — 43 tests

Serializers for DiaApplication manifest, spatial grids (BVH, quadtree, hex grid), and IK chain definition. ApplicationManifestLoader parse bug found and fixed during testing.

Gap-fill session

96 additional tests added across Observer, Graph, FilePath, Easing, FloatMaths boundaries, container stress — filling coverage gaps identified by /gen-tests.

The AI Story

What Claude specifically contributed

Test generation

/gen-tests reads public API headers and generates all 6 test types: unit, golden, invariant, determinism, stress, integration. Tests aren't a chore — they're a command.

Bug discovery

The DiaAnimation2D pointer stability bug. The HexGrid coordinate system error. The Circle closest-point offset bug. All caught by exhaustive tests generated before the code shipped.

Context persistence

Research funnels

Each plan task runs as an independent subagent. Independent tasks run in parallel. DiaAnimation2D, DiaRig2D, and DiaIK2D were specced and partially implemented concurrently.

Quality Gates

Bugs caught before they could ship

Exhaustive test suites — generated from specs, not written as an afterthought — caught real architectural issues during development.

DiaAnimation2D — pointer stability

Spring chain pointers stored raw. When the chain vector reallocated, all pointers dangled. Caught by integration tests that ran the evaluator across multiple frames with growing chain counts.

13 extra tests added, architecture fixed.

HexGrid — axial coordinate system

Grid extent was computed with incorrect coordinate assumptions — grid appeared correct for small sizes but broke at the boundary. Corner-sampling logic was wrong.

Rewritten with true axial coords and new boundary tests.

Circle closest-point — missing offset

Closest-point methods on Circle were computing against the origin, ignoring the circle's center position. Would have produced wrong physics contact normals silently.

Geometry2D edge case tests added and fix committed.

ApplicationManifestLoader — parse duplication

Manifest entries were parsed twice due to a loop structure bug. Only visible under stress tests with large manifests.

9 extra tests added covering duplicate-entry detection.

The Bigger Picture

This isn't just about speed

Architecture that holds

Binding decisions set once at the Platform level flow into every feature — automatically. A decision made in April is still respected in every new module added in December. The AI carries it forward.

Tests as documentation

Golden value tests document expected behavior. Invariant tests document what must always be true. Determinism tests document the simulation contract. Tests are the spec made executable.

Research that's permanent

docs/research/ archives every major decision: why PBD over impulse-based physics, why deps.json over a package manager, why alpine/react over Webix. Future sessions can query the reasoning.

Scale without drift

37+ modules and no two have inconsistent patterns. Every module follows the same naming, every interface follows the same contract, every test follows the same fixture structure. Enforced — not hoped for.

The architecture doesn't survive because people remember it. It survives because the AI reads CLAUDE.md every session.
