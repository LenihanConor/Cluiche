# System Spec: DiaEnv

## Parent Application
@docs/specs/applications/dia.md

## Purpose

DiaEnv is the portable development environment system for the Cluiche platform. It owns the full lifecycle of provisioning a new development machine: declaring which binary SDKs belong in `External/`, declaring which toolchain packages are required, automating their download and installation, verifying environment health, migrating source-only deps to git submodules, integrating dep-restore into the MSBuild pipeline, and ensuring `.claude/` AI context is correctly wired on a new machine.

The core problem DiaEnv solves: a fresh `git clone` of Cluiche produces a source tree that cannot build without a manual, undocumented setup ritual. DiaEnv makes `git clone` → working build → continuing AI engagement a single command.

DiaEnv lives entirely in `Dia/DiaCLI/` as a set of `dia env` sub-commands and supporting config files. It does not introduce a new C++ module.

## Responsibilities

- **SDK Dependency Manifest** — Own `deps.json` at the repo root: a machine-readable, version-pinned, SHA-256-verified list of every binary SDK gitignored from `External/`
- **Toolchain Manifest** — Own `winget.json` at the repo root: a `winget export`-compatible manifest declaring VS 2022 (with correct workloads), Python 3.11, and Git
- **Environment Bootstrap** — `dia env setup`: read both manifests, download/unzip SDKs, invoke `winget import` for toolchain, configure PATH
- **Environment Verification** — `dia env verify`: inspect the current machine against both manifests and report pass/warn/fail with actionable fix commands; exits non-zero on hard failures
- **Source Dep Migration** — Convert source-only, header-only `External/` deps (googletest, pybind11, websocketpp, asio) to git submodules pinned at exact commits
- **MSBuild Auto-Restore** — `RestoreExternals` MSBuild target in `Directory.Build.targets`: reads `deps.json`, downloads any missing SDK entries before compilation; no-op when all deps are present
- **AI Context Hardening** — `dia env claude-setup`: generate `settings.local.json` from a committed template, and symlink the in-repo memory directory to the user-profile Claude Code path

## Non-Responsibilities

- **Runtime game logic** — DiaEnv is development tooling only
- **CI/CD pipeline configuration** — `dia env verify` can be used in CI, but DiaEnv does not own the CI pipeline itself
- **Cross-platform support** — Windows 11 x64 only (PD-005)
- **Ongoing dependency updates** — DiaEnv provides the mechanism to pin and fetch; keeping manifests up to date is a developer responsibility
- **IDE configuration** — VS Code settings, JetBrains config, etc. are out of scope
- **`dia env doctor` (continuous monitoring)** — deferred to a future phase; premature until setup + verify are stable

## Public Interfaces

### CLI Commands

```bash
# Full bootstrap — provision a fresh machine end-to-end
dia env setup                          # run all sub-steps in order
dia env setup --deps-only              # download/unzip SDKs from deps.json only
dia env setup --toolchain-only         # run winget import for VS + Python + Git
dia env setup --claude-only            # configure .claude/ context
dia env setup --dep <name>             # re-download a single named dep

# Environment health check (read-only, CI-safe)
dia env verify                         # check all components; exit 0=ok, 1=fail
dia env verify --json                  # machine-readable output

# AI context wiring
dia env claude-setup                   # generate settings.local.json + symlink memory dir

# Snapshot transfer — deferred; covered by deps.json + file:// mirrors + git submodules
```

### Config Files (repo root)

**`deps.json`** — Binary SDK manifest:
```json
{
  "schema": "diaenv.deps.v1",
  "deps": [
    {
      "id": "ultralight",
      "version": "1.3.0",
      "url": "https://...",
      "sha256": "abc123...",
      "unzip_to": "External/Ultralight",
      "strip_root": true
    }
  ]
}
```

**`winget.json`** — Toolchain manifest (generated by `winget export`, committed to repo):
```json
{
  "$schema": "https://aka.ms/winget-packages.schema.2.0.json",
  "Sources": [ { "Packages": [ ... ] } ]
}
```

**`settings.local.template.json`** — Template for `.claude/settings.local.json`:
```json
{
  "permissions": { "allow": [ ... ], "deny": [] },
  "hooks": { ... }
}
```

### Events Emitted (DiaCLI event system)

- **OnDepDownloadStarted(depId, url)** — before each SDK download begins
- **OnDepDownloadCompleted(depId, path, durationMs)** — after successful unzip
- **OnDepDownloadFailed(depId, url, error)** — on any download or integrity failure
- **OnEnvVerifyCompleted(passCount, warnCount, failCount)** — after verify finishes

### Exit Codes

Follows DiaCLI conventions (SD-CLI-008):
- `0` — All checks passed / setup completed
- `1` — Hard failure (missing dep, download error, integrity mismatch)
- `2` — Warnings only (soft mismatches, non-blocking)
- `3` — Manifest not found or malformed

## Platform Primitives Used

**Python Packages:**
- **click** — CLI argument parsing (existing DiaCLI dependency)
- **requests** + **requests-toolbelt** — HTTP download and streaming (existing in `software_installer`)
- **hashlib** (stdlib) — SHA-256 integrity verification
- **zipfile** / **shutil** (stdlib) — archive extraction
- **loguru** — internal/debug logging (existing DiaCLI dependency)
- **OutputContext** (`dia_cli/utils/dia_output.py`) — structured terminal output and NDJSON event log; all user-facing events emitted via this layer (see `cli-output` feature)

**System Tools:**
- **winget** — Windows Package Manager (ships with Windows 11)
- **vswhere.exe** — VS workload detection (ships with VS 2022 installer, also available standalone)
- **git** — submodule management

**MSBuild:**
- `Directory.Build.targets` — inline PowerShell task for `RestoreExternals`

## Dependencies on Other Systems

**Within DiaCLI (existing primitives DiaEnv extends):**
- `dia_cli/utils/software_installer.py` — download + unzip (DiaEnv extends with manifest + hash verification)
- `dia_cli/utils/dia_cli_config.py` — config file pattern (deps.json follows same schema conventions)
- `dia_cli/utils/dia_cli_paths.py` — path resolution for `External/`
- `dia_cli/commands/setup/setpath_cmd.py` — PATH configuration (DiaEnv builds on this for verify)
- `dia_cli/utils/dia_output.py` (`cli-output` feature) — all user-facing events and terminal output

**External:**
- `Directory.Build.props` — DiaEnv's RestoreExternals target must not override any properties owned by this file (PD-008)

## Out of Scope

- Docker / containerised builds for interactive dev (GPU + GUI) — Hyper-V VM required for that; headless build + test is in scope via the `docker-build-env` feature
- `dia env doctor` continuous monitoring — deferred; premature until setup + verify are stable
- Cloud storage / sync — not required for single-developer use case
- Non-Windows toolchain support — PD-005 enforces Windows 11 x64 only

## Test Strategy

DiaEnv logic is tested at two levels:

### Unit tests (`dia test cli`)
All Python modules introduced by DiaEnv features are covered by pytest unit tests in `Dia/DiaCLI/tests/`. Tests use `unittest.mock` to stub network calls, filesystem writes, and subprocess invocations — no real downloads, no real installs. See `cli-unit-tests` feature spec (@docs/specs/features/dia/diatest/cli-unit-tests.md).

Key modules under test:
- `deps_restore.py` — sentinel logic, SHA-256, mirror fallback, `file://` dispatch
- `toolchain_verify.py` — vswhere, tool version checks, Docker mode detection
- `submodule_verify.py` — git submodule status parsing
- `setup_orchestrator.py` — step ordering, elevation handling, `--fail-fast`
- `verify_orchestrator.py` — PASS/WARN/FAIL aggregation, exit codes, JSON output
- `claude_context_cmd.py` — template generation, symlink creation

### Integration test (`dia test env-integration`)
End-to-end validation inside a fresh Docker container: `dia env docker` → `dia pipeline` → `dia test cli`. The AI fixes environment/infrastructure failures and loops until green. See `env-integration` feature spec (@docs/specs/features/dia/diatest/env-integration.md).

This is the definitive "did DiaEnv work on a clean machine?" test.

## Decisions

| ID | Decision | Rationale | Scope | Status | Binding |
|----|----------|-----------|-------|--------|---------|
| SD-ENV-001 | `deps.json` is the single source of truth for binary SDK versions | Replaces tribal knowledge; enables automation; version-pinned with SHA-256 | All features | Accepted | Yes |
| SD-ENV-002 | SHA-256 hash verification required for all downloaded deps | Prevents corruption and silent version drift | deps-manifest, env-setup-cmd, msbuild-restore | Accepted | Yes |
| SD-ENV-003 | `dia env verify` is read-only and CI-safe | Enables use in pipelines without side effects; exits non-zero on failure | env-verify-cmd | Accepted | Yes |
| SD-ENV-004 | `winget.json` generated by `winget export` and committed to repo | Re-exportable on toolchain change; standard format; no custom schema to maintain | winget-manifest | Accepted | Yes |
| SD-ENV-005 | MSBuild RestoreExternals uses sentinel files to guard against re-download | Build time must not regress when deps are already present | msbuild-restore | Accepted | Yes |
| SD-ENV-006 | `settings.local.json` is never committed; a template is committed instead | Machine-specific settings (paths, hooks) must not be shared verbatim across machines | claude-context | Accepted | Yes |
| SD-ENV-007 | Source-only deps migrated to git submodules; binary SDKs use deps.json | Clean split: git handles source deps automatically; manifest handles binary SDKs | submodule-migration | Accepted | Yes |
| SD-ENV-008 | Docker container scope is headless build + test only; no GPU/GUI | SFML and CluicheEditor require a windowing system; containerised rendering requires Hyper-V VM (future); Windows WARP software renderer deferred | docker-build-env | Accepted | Yes |
| SD-ENV-009 | Docker files live under `build-env/` at repo root | Keeps all environment provisioning artefacts co-located; consistent with research decision | docker-build-env | Accepted | Yes |
| SD-ENV-010 | Python 3.11 is the single Python version for all contexts | Unifies host tooling (DiaCLI) and embedded runtime (`External/Python311`); eliminates version drift between development and runtime; `.python-version` updated to 3.11 | All features | Accepted | Yes |

**Status values:** `Proposed` · `Accepted` · `Rejected` · `Superseded`
**Binding:** `Yes` = enforced constraint on all features in this system · `No` = guidance only

## AI Review Questions

| # | Section | Question | Answer |
|---|---------|----------|--------|
| 1 | Scope | Should DiaEnv be a system under DiaCLI or a standalone application? | System under DiaCLI — it's development tooling with no runtime component, and DiaCLI's plugin architecture is the right home. |
| 2 | deps-manifest | What happens when a download URL returns a 404 (SDK vendor moves CDN)? | `dia env setup` fails fast with a clear error naming the dep and URL. The manifest should include an optional `fallback_url` field for resilience. TBD in feature spec. |
| 3 | winget-manifest | `winget import` is not fully idempotent — what if VS is already installed but missing a workload? | `dia env verify` detects the gap via vswhere.exe; `dia env setup --toolchain` re-invokes the VS installer in modify mode. Full remediation flow to be specced in winget-manifest feature. |
| 4 | submodule-migration | Developers often forget `git submodule update --init` after clone — how is this mitigated? | `dia env verify` checks for uninitialised submodules and reports them. The README will include a `git clone --recurse-submodules` note. Optionally, a post-clone hook can auto-run verify. |
| 5 | msbuild-restore | Could the RestoreExternals target significantly slow incremental builds? | Sentinel files (one `.restored` file per dep) guard against re-download checks on every build. The target only runs network I/O when a sentinel is absent. Must be benchmarked and confirmed in the feature spec. |
| 6 | claude-context | The memory directory path is user-profile-based (`C:\Users\<user>\.claude\projects\...`) — how is the correct path determined? | Claude Code's memory path is derived from the absolute repo path. `dia env claude-setup` computes the expected profile path and creates the symlink. The template generator must handle the path dynamically, not hardcode a username. |
| 7 | env-export | Is `dia env export` needed if all deps can be re-downloaded from canonical URLs? | Only if internet access is slow or restricted. Treat as optional/low-priority. If a dep's URL is unavailable, the export artefact is the only recovery path — so it has value as a backup, not a primary workflow. |
| 8 | Decisions | Should `deps.json` support multiple mirror URLs per dep? | Yes — add an optional `mirrors` array alongside `url`. `dia env setup` tries mirrors in order on 404. Spec this in the deps-manifest feature. |
