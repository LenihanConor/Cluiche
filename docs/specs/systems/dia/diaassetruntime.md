## Summary

DiaAssetRuntime is the in-game asset path resolution and lifecycle management system. It loads `assets.runtime.json` (the deployed manifest generated by DiaAssetPipeline, located at `bin/<App>/<Config>/<Platform>/assets/assets.runtime.json`) at startup, provides ID-to-absolute-path resolution for all known assets, and manages Stage-level load/unload state across Stage transitions. It does not load asset content â€” that remains the responsibility of the consuming system (DiaGraphics, DiaAudio, etc.). It notifies listeners when asset state changes so those systems can respond.

DiaAssetRuntime is a pure C++ library with no dependency on DiaApplicationFlow. It is designed to be wired in as a `Module` by game code, but it has no knowledge of that wiring.

DiaAssetRuntime answers: "where is this asset, is it currently available, and when should it be loaded or released?"

## Responsibilities

**Owns:**
- Manifest loading â€” deserializes `assets.runtime.json` (deployed by DiaAssetPipeline) via `RuntimeManifestLoader` into a local asset table at startup. This manifest contains deploy-relative paths and stage membership â€” no source paths, content hashes, tags, or type metadata.
- Path resolution â€” resolves a StringCRC asset ID to an absolute `FilePath` rooted at the `bin/<App>/<Config>/<Platform>/assets/` deploy directory
- Asset state machine â€” tracks each asset through `Registered â†’ Staged â†’ Loaded â†’ Unloading` states
- Stage-level load/unload requests â€” game code calls `RequestStageLoad(stageId)`; DiaAssetRuntime expands Stage â†’ Assets via `RuntimeStageEntry` and manages state transitions
- Reference counting for shared assets â€” `global/` assets referenced by multiple Stages are not unloaded until all referencing Stages release them
- Handler dispatch â€” invokes registered `IAssetTypeHandler` implementations (keyed by asset type prefix) to perform actual load/unload, tracks completion via `IAssetLoadCallback`
- State query API â€” debug tools and editors query state via `GetAssetState`, `GetLoadedAssets`, etc. and DiaAPI commands over WebSocket
- Debug query API â€” "what's loaded?", "what's pending?", "what does this Stage depend on?"
- Additive path aliasing â€” integrates with DiaCore FilePath's path alias system; asset IDs become a first-class alias source alongside existing path store entries. All resolved paths are absolute.

**Does NOT own:**
- Asset content loading â€” textures, audio, meshes are loaded by DiaGraphics, DiaAudio, etc. in response to events
- DiaApplicationFlow lifecycle â€” no `Startup()`/`Update()`/`Shutdown()` defined here; game code wires that
- Manifest authoring or editing â€” owned by DiaAssetCatalogueEditor
- Asset pipeline or transform â€” owned by DiaAssetPipeline
- Binary format of deployed assets

## Asset State Machine

```
Null         â€” not yet known to the runtime (pre-manifest)
     â”‚
     â”‚  manifest load
     â–Ľ
Staged       â€” known to registry; path resolvable; eligible for loading
     â”‚
     â”‚  RequestStageLoad() â€” handler invoked
     â–Ľ
Loading      â€” IAssetTypeHandler::Load called; I/O in progress
     â”‚
     â”śâ”€â”€[handler success]â”€â”€> Loaded    â€” content confirmed in memory; fully active
     â”‚
     â””â”€â”€[handler failure]â”€â”€> Failed    â€” load attempt failed; blocks IsLoadComplete
                                â”‚
                                â”śâ”€â”€[explicit retry]â”€â”€> Loading
                                â”‚
                                â””â”€â”€[RequestStageUnload]â”€â”€> Unloaded

Loaded       â€” content in memory; fully active
     â”‚
     â”‚  RequestStageUnload() (ref count reaches zero)
     â–Ľ
Unloaded     â€” content released; path still resolvable; eligible for re-staging
     â”‚
     â”‚  re-staging (RequestStageLoad again)
     â–Ľ
Staged       â€” (cycle restarts)
```

**ASSERT:** `RequestStageUnload` while any asset is `Loading` is a programming error â€” asserts in debug builds.

**Reference counting:** Global assets (scope = `global`) are ref-counted across Stage requests. An asset only transitions to `Unloaded` when its ref count drops to zero. Stage-scoped assets have a ref count of 1 â€” owned by their Stage.

**Handler dispatch:** Asset type is derived from the asset ID prefix (before first `.`). A registered `IAssetTypeHandler` for that type prefix performs the actual load/unload. See Feature 7 (Asset Lifecycle Management) for full design.

### Runtime Manifest Format (`assets.runtime.json`)

Generated by DiaAssetPipeline, deployed to `bin/<App>/<Config>/<Platform>/assets/assets.runtime.json`. Deserialized by `RuntimeManifestLoader`.

```json
{
    "assets": [
        {
            "id": "texture.player_ship",
            "scope": "stage",
            "deploy_path": "stages/Gameplay/characters/player_ship.texture.png"
        },
        {
            "id": "entity.player_ship",
            "scope": "stage",
            "deploy_path": "stages/Gameplay/characters/player_ship.entity.json"
        },
        {
            "id": "config.ship_stats",
            "scope": "global",
            "deploy_path": "global/gameplay/ship_stats.config.json"
        },
        {
            "id": "folder.main_ui",
            "scope": "global",
            "deploy_path": "global/Presentation/UI/main_ui.folder/"
        }
    ],
    "stages": [
        {
            "id": "stage.gameplay",
            "assets": [
                "texture.player_ship",
                "entity.player_ship",
                "config.ship_stats",
                "folder.main_ui"
            ]
        },
        {
            "id": "stage.main_menu",
            "assets": [
                "config.ship_stats",
                "folder.main_ui"
            ]
        }
    ]
}
```

**Fields:**
- `assets[].id` â€” StringCRC asset ID (`type.name` format)
- `assets[].scope` â€” `"global"` or `"stage"` (informational â€” the pipeline already placed the file in the correct directory)
- `assets[].deploy_path` â€” path relative to the `assets/` deploy root. `RuntimeManifestLoader` resolves this to an absolute path by prepending the deploy root. Folder assets have a trailing `/`.
- `stages[].id` â€” StringCRC stage ID
- `stages[].assets` â€” array of asset IDs belonging to this stage (derived from `contains` relationships in the catalogue at pipeline time). Global assets that appear in multiple Stages are listed in each Stage that uses them.

**Not present:** source paths, content hashes, tags, type metadata, relationships. This is a lookup table, not a catalogue.

### Stage File Format (`*.stage.json`)

Stage files contain metadata only. Asset membership is defined by `contains` relationships in the catalogue (SD-CAT-012), not in the stage file.

```json
{
    "name": "gameplay",
    "display_name": "Gameplay"
}
```

### Core API

```cpp
namespace Dia::AssetRuntime
{
    enum class AssetState
    {
        Null,
        Staged,
        Loading,
        Loaded,
        Failed,
        Unloaded
    };

    class IAssetLoadCallback
    {
    public:
        virtual void OnLoadComplete(const Dia::Core::StringCRC& assetId) = 0;
        virtual void OnLoadFailed(const Dia::Core::StringCRC& assetId, const char* reason) = 0;
    };

    class IAssetTypeHandler
    {
    public:
        virtual ~IAssetTypeHandler() = default;
        virtual void Load(const Dia::Core::StringCRC& assetId,
                          const Dia::Core::Containers::String512& resolvedPath,
                          IAssetLoadCallback* callback) = 0;
        virtual void Unload(const Dia::Core::StringCRC& assetId) = 0;
    };

    // IAssetStateListener removed â€” handlers own load/unload; debug tools use
    // GetAssetState / DiaAPI commands for observation.

    class AssetRuntime
    {
    public:
        // Startup â€” call once before any queries
        bool LoadManifest(const Dia::Core::FilePath& manifestPath);

        // Path resolution â€” returns String512* (not FilePath*; FilePath's 32-char component
        // limit cannot hold arbitrary absolute paths)
        const Dia::Core::Containers::String512* ResolveAssetPath(const Dia::Core::StringCRC& assetId) const;

        // Stage lifecycle â€” expands Stage â†’ Assets
        void RequestStageLoad(const Dia::Core::StringCRC& stageId);
        void RequestStageUnload(const Dia::Core::StringCRC& stageId);

        // Type handler registration (keyed by asset ID prefix)
        void RegisterTypeHandler(const char* typePrefix, IAssetTypeHandler* handler);
        void UnregisterTypeHandler(const char* typePrefix);

        // Explicit retry for failed assets
        void RetryAssetLoad(const Dia::Core::StringCRC& assetId);

        // Load progress (loaded/total/failed counts for a stage)
        struct LoadProgress { unsigned int loaded; unsigned int total; unsigned int failed; };
        LoadProgress GetLoadProgress(const Dia::Core::StringCRC& stageId) const;

        // State queries
        AssetState GetAssetState(const Dia::Core::StringCRC& assetId) const;
        bool IsAssetReady(const Dia::Core::StringCRC& assetId) const;

        // Type handler registration (see above)
        // IAssetStateListener removed â€” debug tools use query API / DiaAPI commands

        // Teardown â€” fire OnAssetUnloading for all non-Registered assets, reset state
        void Reset();

        // Debug queries â€” returns total count (may exceed array capacity)
        unsigned int GetAllAssets(
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
        unsigned int GetLoadedAssets(
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
        unsigned int GetStagedAssets(
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
        unsigned int GetStageDependencies(
            const Dia::Core::StringCRC& stageId,
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;

        // Debug info
        unsigned int GetAssetRefCount(const Dia::Core::StringCRC& assetId) const;
        AssetScope GetAssetScope(const Dia::Core::StringCRC& assetId) const;
        Dia::Core::StringCRC GetAssetStageId(const Dia::Core::StringCRC& assetId) const;
    };
}
```

### Usage Pattern

```cpp
// Startup â€” register handlers by asset type prefix
Dia::AssetRuntime::AssetRuntime runtime;
runtime.LoadManifest(manifestPath);
runtime.RegisterTypeHandler("texture", &myTextureHandler);
runtime.RegisterTypeHandler("ui", &myUIHandler);

// Stage transition â€” dispatches Load to handlers for each asset
runtime.RequestStageLoad(StringCRC("stage.gameplay"));

// In IAssetTypeHandler::Load â€” system performs actual I/O
void TextureHandler::Load(const StringCRC& assetId, const String512& path,
                          IAssetLoadCallback* callback)
{
    bool ok = mTextures.LoadFromFile(path.AsCStr());
    if (ok)
        callback->OnLoadComplete(assetId);
    else
        callback->OnLoadFailed(assetId);
}

// Path query without lifecycle management
const String512* path = runtime.ResolveAssetPath(StringCRC("config.ship_stats"));

// Stage transition â€” calls Unload on handlers, releases content
runtime.RequestStageUnload(StringCRC("stage.gameplay"));
```

## Design Constraints

- **Pure C++ library, no DiaApplicationFlow dependency.** Game code wires DiaAssetRuntime into a Module. DiaAssetRuntime has no knowledge of ProcessingUnits or Phases.
- **Additive over DiaCore FilePath.** Asset IDs are registered as path aliases; existing game code using FilePath directly is unaffected. All resolved paths are absolute, rooted at the deploy directory.
- **Does not load content directly.** DiaAssetRuntime dispatches to `IAssetTypeHandler` implementations which perform actual I/O. Handlers live in game code, not DiaAssetRuntime â€” the runtime coordinates the lifecycle but owns no format-specific loading logic.
- **Reads `assets.runtime.json`, not `assets.catalogue.json`.** The runtime manifest is generated by DiaAssetPipeline and contains deploy paths and stage membership only â€” no source paths, content hashes, tags, or type metadata.
- **No dependency on DiaAssetCatalogue.** DiaAssetRuntime owns its own data types (`RuntimeAssetEntry`, `RuntimeStageEntry`) and its own deserializer (`RuntimeManifestLoader`). The two manifest formats are independent; the pipeline transforms one into the other. This keeps the game binary free of build-time type framework and authoring machinery.
- **Stage-level granularity.** Game code operates on Stages, not individual assets. Asset-level state is an implementation detail.
- **Reference counting for global assets.** An asset shared by multiple Stages is not unloaded until all referencing Stages release them.
- **No STL containers in public APIs (PD-004, AD-002).** All public interfaces use DiaCore containers.

## AI Review Questions

| # | Section | Question | Answer |
|---|---------|----------|--------|
| 1 | State machine | What happens if `RequestStageLoad` is called for a Stage that is already `Loaded`? | Increment ref counts on its assets, no state change, no events fired. Idempotent for already-loaded assets. |
| 2 | State machine | What happens if `RequestStageUnload` is called while assets are in `Loading` state? | ASSERT in debug builds â€” this is a programming error in the game's flow logic (AC5 of Asset Lifecycle Management feature). Game code must wait for `IsLoadComplete()` or handle failure before unloading a stage. Assets in `Staged` (not yet dispatched to handler) can be unloaded safely. |
| 3 | Listeners | Should listeners be called synchronously on the calling thread, or queued for the next update? | Synchronously for now â€” DiaAssetRuntime is a library with no thread model of its own. If thread safety is needed, the Module wrapper above DiaAssetRuntime owns that synchronization. |
| 4 | Path resolution | Should `ResolveAssetPath` return null for assets not in `Loaded` or `Staged` state, or always return the path if the asset is registered? | Always return the path if the asset is registered â€” path resolution is independent of load state. Game code may need to know the path before requesting a load (e.g. for streaming decisions). Returned path is always absolute, rooted at the deploy directory. |
| 5 | Manifest location | How does DiaAssetRuntime know the manifest path? Is it hardcoded, passed at startup, or resolved via DiaCore FilePath alias? | Passed at startup via `LoadManifest(const FilePath& manifestPath)`. Caller (game code or Module wrapper) resolves the path to `assets.runtime.json` in the deploy directory. No hardcoding in DiaAssetRuntime. |
| 6 | Missing assets | What happens if `ResolveAssetPath` is called with an ID not in the manifest? | Returns null. Caller must check. DiaLogger logs a warning with the unknown ID. No assert â€” missing asset lookups are a runtime condition, not a programming error. |
| 7 | Incremental builds | If DiaAssetPipeline is not run and deployed files are stale, DiaAssetRuntime returns paths to potentially outdated files. Is that acceptable? | Yes â€” DiaAssetRuntime trusts `assets.runtime.json` and the deployed state. It is not responsible for detecting stale assets. That is DiaAssetPipeline's concern (future incremental build feature). |
| 8 | Folder assets | How does DiaAssetRuntime resolve Folder assets? | `ResolveAssetPath` returns the absolute path to the folder directory. The consuming system is responsible for knowing the internal structure. Folder assets transition through the same state machine as file assets. |
