## Summary

DiaAssetRuntime is the in-game asset path resolution and lifecycle management system. It loads `assets.runtime.json` (the deployed manifest generated by DiaAssetPipeline, located at `bin/<App>/<Config>/<Platform>/assets/assets.runtime.json`) at startup, provides ID-to-absolute-path resolution for all known assets, and manages Stage-level load/unload state across Stage transitions. It does not load asset content — that remains the responsibility of the consuming system (DiaGraphics, DiaAudio, etc.). It notifies listeners when asset state changes so those systems can respond.

DiaAssetRuntime is a pure C++ library with no dependency on DiaApplication. It is designed to be wired in as a `Module` by game code, but it has no knowledge of that wiring.

DiaAssetRuntime answers: "where is this asset, is it currently available, and when should it be loaded or released?"

## Responsibilities

**Owns:**
- Manifest loading — deserializes `assets.runtime.json` (deployed by DiaAssetPipeline) via `RuntimeManifestLoader` into a local asset table at startup. This manifest contains deploy-relative paths and stage membership — no source paths, content hashes, tags, or type metadata.
- Path resolution — resolves a StringCRC asset ID to an absolute `FilePath` rooted at the `bin/<App>/<Config>/<Platform>/assets/` deploy directory
- Asset state machine — tracks each asset through `Registered → Staged → Loaded → Unloading` states
- Stage-level load/unload requests — game code calls `RequestStageLoad(stageId)`; DiaAssetRuntime expands Stage → Assets via `RuntimeStageEntry` and manages state transitions
- Reference counting for shared assets — `global/` assets referenced by multiple Stages are not unloaded until all referencing Stages release them
- Event notification — fires `OnAssetReady` / `OnAssetUnloading` events so consuming systems can load/release content
- Debug query API — "what's loaded?", "what's pending?", "what does this Stage depend on?"
- Additive path aliasing — integrates with DiaCore FilePath's path alias system; asset IDs become a first-class alias source alongside existing path store entries. All resolved paths are absolute.

**Does NOT own:**
- Asset content loading — textures, audio, meshes are loaded by DiaGraphics, DiaAudio, etc. in response to events
- DiaApplication lifecycle — no `Startup()`/`Update()`/`Shutdown()` defined here; game code wires that
- Manifest authoring or editing — owned by DiaAssetCatalogueEditor
- Asset pipeline or transform — owned by DiaAssetPipeline
- Binary format of deployed assets

## Asset State Machine

```
Registered   — known to the registry; path resolvable; content not requested
     │
     │  RequestStageLoad()
     ▼
Staged       — load requested; path resolved and available; awaiting consumer acknowledgement
     │
     │  AcknowledgeAssetLoaded()  (called by consumer after content is loaded)
     ▼
Loaded       — content confirmed loaded by consumer; fully active
     │
     │  RequestStageUnload()
     ▼
Unloading    — unload requested; ref count reached zero; OnAssetUnloading fired
     │
     │  AcknowledgeAssetUnloaded()  (called by consumer after content is released)
     ▼
Registered   — back to idle; path still resolvable
```

**Reference counting:** Global assets (scope = `global`) are ref-counted across Stage requests. An asset only transitions to `Unloading` when its ref count drops to zero. Stage-scoped assets have a ref count of 1 — owned by their Stage.

## Public API

### Core Classes

| Class | Responsibility |
|-------|---------------|
| `AssetRuntime` | Central class. Owns the runtime asset table, state machine, ref counts, listener list. |
| `RuntimeAssetEntry` | Lightweight runtime record: StringCRC ID, scope (`global`/`stage`), absolute deploy path. No source path, no content hash, no tags, no type metadata. |

### Runtime Manifest Format (`assets.runtime.json`)

Generated by DiaAssetPipeline, deployed to `bin/<App>/<Config>/<Platform>/assets/assets.runtime.json`. Deserialized by `RuntimeManifestLoader`.

```json
{
    "assets": [
        {
            "id": "texture.player_ship",
            "scope": "stage",
            "deploy_path": "stages/Gameplay/characters/player_ship.texture.png"
        },
        {
            "id": "entity.player_ship",
            "scope": "stage",
            "deploy_path": "stages/Gameplay/characters/player_ship.entity.json"
        },
        {
            "id": "config.ship_stats",
            "scope": "global",
            "deploy_path": "global/gameplay/ship_stats.config.json"
        },
        {
            "id": "folder.main_ui",
            "scope": "global",
            "deploy_path": "global/Presentation/UI/main_ui.folder/"
        }
    ],
    "stages": [
        {
            "id": "stage.gameplay",
            "assets": [
                "texture.player_ship",
                "entity.player_ship",
                "config.ship_stats",
                "folder.main_ui"
            ]
        },
        {
            "id": "stage.main_menu",
            "assets": [
                "config.ship_stats",
                "folder.main_ui"
            ]
        }
    ]
}
```

**Fields:**
- `assets[].id` — StringCRC asset ID (`type.name` format)
- `assets[].scope` — `"global"` or `"stage"` (informational — the pipeline already placed the file in the correct directory)
- `assets[].deploy_path` — path relative to the `assets/` deploy root. `RuntimeManifestLoader` resolves this to an absolute path by prepending the deploy root. Folder assets have a trailing `/`.
- `stages[].id` — StringCRC stage ID
- `stages[].assets` — array of asset IDs belonging to this stage (derived from `contains` relationships in the catalogue at pipeline time). Global assets that appear in multiple Stages are listed in each Stage that uses them.

**Not present:** source paths, content hashes, tags, type metadata, relationships. This is a lookup table, not a catalogue.

### Stage File Format (`*.stage.json`)

Stage files contain metadata only. Asset membership is defined by `contains` relationships in the catalogue (SD-CAT-012), not in the stage file.

```json
{
    "name": "gameplay",
    "display_name": "Gameplay"
}
```

### Core API

```cpp
namespace Dia::AssetRuntime
{
    enum class AssetState
    {
        Registered,
        Staged,
        Loaded,
        Unloading
    };

    class IAssetStateListener
    {
    public:
        virtual void OnAssetReady(const Dia::Core::StringCRC& assetId,
                                  const Dia::Core::FilePath& resolvedPath) = 0;
        virtual void OnAssetUnloading(const Dia::Core::StringCRC& assetId) = 0;
    };

    class AssetRuntime
    {
    public:
        // Startup — call once before any queries
        bool LoadManifest(const Dia::Core::FilePath& manifestPath);

        // Path resolution
        const Dia::Core::FilePath* ResolveAssetPath(const Dia::Core::StringCRC& assetId) const;

        // Stage lifecycle — expands Stage → Assets
        void RequestStageLoad(const Dia::Core::StringCRC& stageId);
        void RequestStageUnload(const Dia::Core::StringCRC& stageId);

        // Consumer acknowledgement
        void AcknowledgeAssetLoaded(const Dia::Core::StringCRC& assetId);
        void AcknowledgeAssetUnloaded(const Dia::Core::StringCRC& assetId);

        // State queries
        AssetState GetAssetState(const Dia::Core::StringCRC& assetId) const;
        bool IsAssetReady(const Dia::Core::StringCRC& assetId) const;

        // Event subscription
        void RegisterListener(IAssetStateListener* listener);
        void UnregisterListener(IAssetStateListener* listener);

        // Debug query
        void GetLoadedAssets(
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
        void GetStagedAssets(
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
        void GetStageDependencies(
            const Dia::Core::StringCRC& stageId,
            Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 128>& results) const;
    };
}
```

### Usage Pattern

```cpp
// Startup
Dia::AssetRuntime::AssetRuntime runtime;
runtime.LoadManifest(manifestPath);
runtime.RegisterListener(&myGraphicsSystem);

// Stage transition — one call loads all Assets in the Stage
runtime.RequestStageLoad(StringCRC("stage.gameplay"));

// In IAssetStateListener::OnAssetReady — consumer loads content
void MyGraphicsSystem::OnAssetReady(const StringCRC& assetId, const FilePath& path)
{
    if (assetId == StringCRC("texture.player_ship"))
        mTextures.Load(assetId, path);
}

// Path query without lifecycle management
const FilePath* path = runtime.ResolveAssetPath(StringCRC("config.ship_stats"));

// Stage transition — unload all Assets owned by this Stage
runtime.RequestStageUnload(StringCRC("stage.gameplay"));
```

## Dependencies

| Dependency | What DiaAssetRuntime uses from it |
|------------|----------------------------------|
| DiaCore — FilePath | Path aliasing — asset IDs registered as aliases in the path store; `FilePath` returned from `ResolveAssetPath` |
| DiaCore — CRC/StringCRC | Asset IDs, Stage IDs |
| DiaCore — Containers | `HashTableC` (asset table, state tracking), `DynamicArrayC` (listener list, query results, stage member lists) |
| DiaCore — JSON | jsoncpp wrapper for deserializing `assets.runtime.json` via `RuntimeManifestLoader` |
| DiaLogger | State transition logging (asset ID, old state, new state, ref count) |

**No dependency on DiaAssetCatalogue.** DiaAssetRuntime owns its own manifest format (`assets.runtime.json`), its own data types (`RuntimeAssetEntry`, `RuntimeStageEntry`), and its own deserializer (`RuntimeManifestLoader`). The two manifest formats are independent — the pipeline (Python) transforms one into the other.

No dependency on DiaApplication, DiaGraphics, DiaSFML, DiaInput, or any rendering/windowing module.

## Design Constraints

- **Pure C++ library, no DiaApplication dependency.** Game code wires DiaAssetRuntime into a Module. DiaAssetRuntime has no knowledge of ProcessingUnits or Phases.
- **Additive over DiaCore FilePath.** Asset IDs are registered as path aliases; existing game code using FilePath directly is unaffected. All resolved paths are absolute, rooted at the deploy directory.
- **Does not load content.** DiaAssetRuntime resolves paths and manages state. Actual bytes-to-memory loading is the consumer's responsibility, triggered by `OnAssetReady`.
- **Reads `assets.runtime.json`, not `assets.catalogue.json`.** The runtime manifest is generated by DiaAssetPipeline and contains deploy paths and stage membership only — no source paths, content hashes, tags, or type metadata.
- **No dependency on DiaAssetCatalogue.** DiaAssetRuntime owns its own data types (`RuntimeAssetEntry`, `RuntimeStageEntry`) and its own deserializer (`RuntimeManifestLoader`). The two manifest formats are independent; the pipeline transforms one into the other. This keeps the game binary free of build-time type framework and authoring machinery.
- **Stage-level granularity.** Game code operates on Stages, not individual assets. Asset-level state is an implementation detail.
- **Reference counting for global assets.** An asset shared by multiple Stages is not unloaded until all referencing Stages release them.
- **No STL containers in public APIs (PD-004, AD-002).** All public interfaces use DiaCore containers.

## Decisions

| ID | Decision | Rationale | Scope | Status | Binding |
|----|----------|-----------|-------|--------|---------|
| SD-ARUN-001 | DiaAssetRuntime has no dependency on DiaApplication | Pure library; game code owns the wiring into ProcessingUnit/Module. Avoids circular or layering constraints. | All features | Accepted | Yes |
| SD-ARUN-002 | Stage is the unit of load/unload; asset is the unit of state | Game code thinks in Stages (coarse); runtime tracks assets (fine). Matches how content systems (DiaGraphics) respond per-asset. | All features | Accepted | Yes |
| SD-ARUN-003 | Global assets are ref-counted; Stage-scoped assets are not | Global assets are shared — must not unload while any Stage still uses them. Stage-scoped assets are owned by one Stage — ref count is always 0 or 1. | All features | Accepted | Yes |
| SD-ARUN-004 | Consumers acknowledge load/unload completion | DiaAssetRuntime fires the event and waits for acknowledgement before advancing state. Prevents state desync between runtime and content systems. | All features | Accepted | Yes |
| SD-ARUN-005 | Asset IDs registered as path aliases in DiaCore FilePath | Additive integration — existing FilePath users unaffected; asset-aware code can resolve by ID without knowing the deploy layout. | All features | Accepted | Yes |
| SD-ARUN-006 | No content loading in DiaAssetRuntime | Single responsibility — path resolution and lifecycle management only. Keeps DiaAssetRuntime free of DiaGraphics/DiaAudio dependencies. | All features | Accepted | Yes |

## AI Review Questions

| # | Section | Question | Answer |
|---|---------|----------|--------|
| 1 | State machine | What happens if `RequestStageLoad` is called for a Stage that is already `Loaded`? | Increment ref counts on its assets, no state change, no events fired. Idempotent for already-loaded assets. |
| 2 | State machine | What happens if `RequestStageUnload` is called before consumers have acknowledged all `OnAssetReady` events (asset in `Staged` state)? | Transition to `Unloading` immediately; fire `OnAssetUnloading`. Consumer must handle receiving `OnAssetUnloading` before they finished loading — they should abort their load and call `AcknowledgeAssetUnloaded`. |
| 3 | Listeners | Should listeners be called synchronously on the calling thread, or queued for the next update? | Synchronously for now — DiaAssetRuntime is a library with no thread model of its own. If thread safety is needed, the Module wrapper above DiaAssetRuntime owns that synchronization. |
| 4 | Path resolution | Should `ResolveAssetPath` return null for assets not in `Loaded` or `Staged` state, or always return the path if the asset is registered? | Always return the path if the asset is registered — path resolution is independent of load state. Game code may need to know the path before requesting a load (e.g. for streaming decisions). Returned path is always absolute, rooted at the deploy directory. |
| 5 | Manifest location | How does DiaAssetRuntime know the manifest path? Is it hardcoded, passed at startup, or resolved via DiaCore FilePath alias? | Passed at startup via `LoadManifest(const FilePath& manifestPath)`. Caller (game code or Module wrapper) resolves the path to `assets.runtime.json` in the deploy directory. No hardcoding in DiaAssetRuntime. |
| 6 | Missing assets | What happens if `ResolveAssetPath` is called with an ID not in the manifest? | Returns null. Caller must check. DiaLogger logs a warning with the unknown ID. No assert — missing asset lookups are a runtime condition, not a programming error. |
| 7 | Incremental builds | If DiaAssetPipeline is not run and deployed files are stale, DiaAssetRuntime returns paths to potentially outdated files. Is that acceptable? | Yes — DiaAssetRuntime trusts `assets.runtime.json` and the deployed state. It is not responsible for detecting stale assets. That is DiaAssetPipeline's concern (future incremental build feature). |
| 8 | Folder assets | How does DiaAssetRuntime resolve Folder assets? | `ResolveAssetPath` returns the absolute path to the folder directory. The consuming system is responsible for knowing the internal structure. Folder assets transition through the same state machine as file assets. |
