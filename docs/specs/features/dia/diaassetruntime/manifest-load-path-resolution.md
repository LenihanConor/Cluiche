## Summary

`RuntimeManifestLoader` deserializes `assets.runtime.json` into `RuntimeAssetEntry` and `RuntimeStageEntry` tables stored in `HashTableC` containers. `ResolveAssetPath` returns the absolute deploy path for any registered asset ID. Asset IDs are registered as path aliases in DiaCore's `PathStore`. The loader is standalone with no dependency on DiaAssetCatalogue.

This is the bootstrap feature for DiaAssetRuntime -- every other feature depends on the asset and stage tables it populates. The manifest is generated by DiaAssetPipeline (a Python pipeline) and deployed alongside game binaries. DiaAssetRuntime only reads it.

## Problem

Game code needs to resolve asset IDs to absolute file paths at runtime without knowing the deploy directory layout. Currently there is no system that reads the deployed manifest, resolves relative paths to absolute paths, or registers asset IDs as path aliases. Every consuming system (DiaGraphics, DiaAudio) would need to hardcode or independently discover file locations. A single manifest loader with path resolution and alias registration solves this.

## Acceptance Criteria

1. `RuntimeManifestLoader::Load` reads a JSON file at the given `FilePath`, parses it into `RuntimeAssetEntry` and `RuntimeStageEntry` records, and populates caller-provided `HashTableC` containers
2. `RuntimeAssetEntry` stores: StringCRC `mId`, `AssetScope mScope` (kGlobal or kStage), `FilePath mDeployPath` (absolute, resolved from deploy root)
3. `RuntimeStageEntry` stores: StringCRC `mId`, `DynamicArrayC<StringCRC, 64> mAssetIds`
4. `AssetScope` is an enum with values `kGlobal` and `kStage`
5. `deploy_path` values in the JSON are relative to the manifest file's parent directory; the loader resolves them to absolute paths by prepending that parent directory
6. Folder assets have a trailing `/` in their `deploy_path` -- the loader preserves this
7. Duplicate asset IDs within the manifest are rejected with a clear error; loading fails and reports which ID is duplicated
8. Duplicate stage IDs within the manifest are rejected with a clear error
9. `ResolveAssetPath` on the `AssetRuntime` class returns a `const FilePath*` for a given asset ID, or null if the ID is not registered
10. All asset IDs are registered as path aliases in DiaCore's `PathStore` after loading, so downstream code can resolve assets via the path alias system
11. No dependency on DiaAssetCatalogue -- the loader owns its own types and parsing logic
12. No file watching, no caching, no content loading -- pure manifest parsing and path resolution

## API Design

### Core Types

```cpp
namespace Dia::AssetRuntime
{
    enum class AssetScope
    {
        kGlobal,
        kStage
    };

    struct RuntimeAssetEntry
    {
        Dia::Core::StringCRC mId;
        AssetScope mScope;
        Dia::Core::FilePath mDeployPath; // absolute, resolved from deploy root
    };

    struct RuntimeStageEntry
    {
        Dia::Core::StringCRC mId;
        Dia::Core::Containers::DynamicArrayC<Dia::Core::StringCRC, 64> mAssetIds;
    };

    class RuntimeManifestLoader
    {
    public:
        bool Load(const Dia::Core::FilePath& manifestPath,
                  Dia::Core::Containers::HashTableC<Dia::Core::StringCRC, RuntimeAssetEntry, 512>& assetTable,
                  Dia::Core::Containers::HashTableC<Dia::Core::StringCRC, RuntimeStageEntry, 32>& stageTable);
    };
}
```

### Runtime Manifest Format

```json
{
    "assets": [
        {
            "id": "texture.player_ship",
            "scope": "stage",
            "deploy_path": "stages/Gameplay/characters/player_ship.texture.png"
        },
        {
            "id": "config.ship_stats",
            "scope": "global",
            "deploy_path": "global/gameplay/ship_stats.config.json"
        },
        {
            "id": "folder.main_ui",
            "scope": "global",
            "deploy_path": "global/Presentation/UI/main_ui.folder/"
        }
    ],
    "stages": [
        {
            "id": "stage.gameplay",
            "assets": ["texture.player_ship", "config.ship_stats", "folder.main_ui"]
        }
    ]
}
```

### Path Resolution API

```cpp
namespace Dia::AssetRuntime
{
    class AssetRuntime
    {
    public:
        bool LoadManifest(const Dia::Core::FilePath& manifestPath);
        const Dia::Core::FilePath* ResolveAssetPath(const Dia::Core::StringCRC& assetId) const;
    };
}
```

### Usage Pattern

```cpp
Dia::AssetRuntime::AssetRuntime runtime;
runtime.LoadManifest(manifestPath);

// Resolve by ID
const Dia::Core::FilePath* path = runtime.ResolveAssetPath(StringCRC("texture.player_ship"));
// path is absolute: "C:/Game/bin/Debug/x64/assets/stages/Gameplay/characters/player_ship.texture.png"
```

## AI Review Questions

| # | Section | Question | Answer |
|---|---------|----------|--------|
| 1 | Manifest | What happens if the manifest file does not exist or cannot be opened? | `RuntimeManifestLoader::Load` returns false. No partial state -- the asset and stage tables remain empty. DiaLogger logs the file path and the error. |
| 2 | Path resolution | Should the loader validate that resolved absolute paths actually exist on disk? | No. The loader trusts the manifest. Path existence checking is the consuming system's responsibility when it attempts to load content. This avoids I/O overhead at startup and supports streaming scenarios where files may not yet be deployed. |
| 3 | Capacity | HashTableC has a fixed capacity of 512 assets and 32 stages. What if the manifest exceeds these? | Load fails and returns false with a DiaLogger error. For a small/medium game these capacities are generous. If a project needs more, the template parameters can be increased at the call site. |
| 4 | Scope parsing | What if the `scope` field in JSON contains an unrecognized value (not "global" or "stage")? | Reject the entry with a DiaLogger warning and fail the load. Unknown scope values indicate a manifest version mismatch or corruption. |
| 5 | Alias registration | If an asset ID conflicts with an existing PathStore alias, what happens? | The asset ID alias overwrites the existing alias with a DiaLogger warning. Asset manifest is authoritative for asset paths. |
| 6 | Folder assets | How does the loader distinguish folder assets from file assets? | By the trailing `/` in `deploy_path`. The loader preserves this in the FilePath. Consuming systems check for a trailing separator to know they are dealing with a directory. |
