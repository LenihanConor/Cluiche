# Feature Spec: winget-manifest

## Parent System
@docs/specs/systems/dia/diaenv.md

## Status
`Done`

## Summary

Define and commit `winget.json` — a `winget export`-compatible toolchain manifest declaring every developer tool required to build and run Cluiche that is not a binary SDK in `External/`. Implement `dia env setup --toolchain` to invoke `winget import` from this manifest, and extend `dia env verify` to check each tool's presence and version. Covers: Visual Studio 2022 (C++ Desktop workload), Python 3.11 (host, for DiaCLI — unified with embedded runtime per SD-ENV-010), Git, Node.js LTS (for React UI builds), Poetry (DiaCLI dependency management), and Docker Desktop (Windows Containers mode).

## Problem

There is no machine-readable record of which developer tools are required to build Cluiche. A fresh Windows install requires manually tracking down and installing VS 2022, Python, Git, Node.js, Poetry, and Docker Desktop with the correct versions. Without this manifest, toolchain drift between machines causes hard-to-reproduce build failures.

## Goals

- Commit `winget.json` to repo root covering all 6 required tools
- `dia env setup --toolchain` installs all tools via `winget import` in one command
- Elevation check before running `winget import` (requires admin)
- VS-closed check before installing VS 2022 workloads (installer fails if VS is open)
- `dia env verify` reports presence and version for each tool: PASS / WARN / FAIL
- VS workload gap detection via `vswhere.exe` — detect if VS is installed but missing C++ Desktop workload

## Non-Goals

- Embedded Python 3.11 runtime (`External/Python311`) — covered by `deps-manifest`
- MkDocs, blue-console tooling — docs and prototype tooling; not required to build the engine
- Automatic tool version upgrades — `winget-manifest` pins versions; updates are a manual developer responsibility
- Non-Windows toolchain support — PD-005 enforces Windows 11 x64 only
- Silent unattended install of VS workloads on a machine where VS is already open — operator must close VS first

## Toolchain Manifest

| Python 3.11 | `Python.Python.3.11` | 3.11.x | Host Python for DiaCLI scripts; unified with embedded runtime version (SD-ENV-010) |

### `winget.json` format

Generated by `winget export` and committed to repo root. Format follows the standard winget packages schema:

```json
{
  "$schema": "https://aka.ms/winget-packages.schema.2.0.json",
  "CreationDate": "2026-04-25",
  "Sources": [
    {
      "Packages": [
        { "PackageIdentifier": "Microsoft.VisualStudio.2022.Community" },
        { "PackageIdentifier": "Python.Python.3.11" },
        { "PackageIdentifier": "Git.Git" },
        { "PackageIdentifier": "OpenJS.NodeJS.LTS" },
        { "PackageIdentifier": "Python-Poetry.Poetry" },
        { "PackageIdentifier": "Docker.DockerDesktop" }
      ],
      "SourceDetails": {
        "Argument": "https://cdn.winget.microsoft.com/cache",
        "Identifier": "Microsoft.Winget.Source_8wekyb3d8bbwe",
        "Name": "winget",
        "Type": "Microsoft.PreIndexed.Package"
      }
    }
  ],
  "WinGetVersion": "1.6.0"
}
```

### VS 2022 workload requirements

`winget import` installs VS 2022 but does not configure workloads. A separate VS installer invocation is needed for workload configuration:

```powershell
# Install C++ Desktop workload (run after winget import)
& "C:\Program Files (x86)\Microsoft Visual Studio\Installer\vs_installer.exe" modify `
  --installPath "C:\Program Files\Microsoft Visual Studio\2022\Community" `
  --add Microsoft.VisualStudio.Workload.NativeDesktop `
  --includeRecommended --quiet --wait
```

This is orchestrated by `dia env setup --toolchain`.

## CLI Interface

```bash
# Install all tools from winget.json
dia env setup --toolchain

# Verify toolchain presence and versions (read-only)
dia env verify
dia env verify --toolchain   # toolchain checks only
```

## Implementation

### Files introduced / modified

```
winget.json                                           # New — toolchain manifest (repo root)
Dia/DiaCLI/
└── dia_cli/
    ├── cli/
    │   └── env/
    │       └── setup_cmd.py                          # Extended — add --toolchain flag
    ├── commands/
    │   └── env/
    │       └── toolchain_install_cmd.py              # New — winget import + VS workload install
    └── utils/
        └── toolchain_verify.py                       # New — vswhere.exe + tool version checks
```

### `toolchain_install_cmd.py` responsibilities

1. Check for admin elevation; exit 1 with clear message if not elevated
2. Check VS not running (look for `devenv.exe` process); warn and prompt to close if found
3. Run `winget import winget.json --accept-package-agreements --accept-source-agreements`
4. After winget completes, invoke VS installer to add C++ Desktop workload if not already present
5. Emit `OnToolchainInstallCompleted(toolCount, durationMs)` event

### `toolchain_verify.py` responsibilities

- **VS 2022**: Use `vswhere.exe` to detect installation path and C++ Desktop workload presence
- **Python 3.11**: `python --version` or registry check
- **Git**: `git --version`
- **Node.js**: `node --version`
- **Poetry**: `poetry --version`
- **Docker Desktop**: Check for `Docker Desktop.exe` process or service; check Windows Containers mode via `docker info`

Reports PASS / WARN / FAIL per tool. WARN = installed but wrong version. FAIL = not installed.

### `vswhere.exe` location

Ships with VS 2022 installer. Also available standalone at:
`%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe`

## Dependencies

| Dependency | Type | Notes |
|------------|------|-------|
| `winget` (Windows Package Manager) | Runtime | Ships with Windows 11; may need updating on older Windows 10 installs |
| `vswhere.exe` | Runtime | Ships with VS 2022 installer; must exist before verify can check VS workloads |
| `env-verify-cmd` feature | Integration | `toolchain_verify.py` is consumed by `dia env verify`; can be developed independently |

## Acceptance Criteria

1. `winget.json` exists at repo root and is valid against the winget packages schema 2.0
2. `dia env setup --toolchain` runs `winget import` and the VS workload installer without user interaction (silent install)
3. `dia env setup --toolchain` exits 1 with a clear error if not run as administrator
4. `dia env setup --toolchain` detects `devenv.exe` running and warns the user before proceeding with VS workload install
5. `dia env verify` reports PASS/WARN/FAIL for each of the 6 tools
6. `dia env verify` detects VS 2022 installed but C++ Desktop workload missing and reports FAIL with an actionable fix command
7. `dia env verify` detects Docker Desktop installed but not in Windows Containers mode and reports WARN
8. `winget.json` is re-generatable by running `winget export -o winget.json` on a correctly provisioned machine

## AI Review Questions

| # | Section | Question | Answer |
|---|---------|----------|--------|
| 1 | VS workload | Does `winget import` install VS 2022 workloads, or only the base VS installer? | `winget import` installs the VS shell only; workloads require a separate `vs_installer.exe modify` invocation. Both steps are orchestrated by `dia env setup --toolchain`. |
| 2 | Poetry install | `winget` does not have a stable Poetry package identifier — should Poetry be installed via `pip install poetry` instead? | Yes — Poetry should be installed via `pip install poetry` after Python 3.11 is confirmed present, rather than via winget. Remove from `winget.json`; add as a post-winget step in `toolchain_install_cmd.py`. |
| 3 | Docker Desktop | Docker Desktop requires Windows Containers mode to be switched manually after install — should `dia env setup --toolchain` prompt the user to switch mode? | Yes — after Docker Desktop install, print a clear instruction: "Switch Docker Desktop to Windows Containers mode: right-click tray icon → Switch to Windows containers". Cannot be automated silently. |
| 4 | Node.js version | `package.json` files specify `>=10` but LTS 20.x is recommended — should a minimum version be enforced in `toolchain_verify.py`? | Yes — warn (WARN) if Node.js < 18.x is detected; fail (FAIL) if < 10.x. 18.x is the oldest LTS still receiving security updates. |
| 5 | Elevation check | Should `dia env setup --toolchain` auto-elevate via UAC, or just exit with an error if not elevated? | Exit with a clear error and print the re-run command prefixed with `Run as Administrator`. Auto-elevation via UAC from Python is fragile and surprising. |
| 6 | Idempotency | If VS 2022 is already installed with the correct workload, does `winget import` re-install it? | `winget import` skips already-installed packages by default. `vswhere.exe` check before the VS workload step means the workload installer is only invoked if the workload is absent. Fully idempotent. |
